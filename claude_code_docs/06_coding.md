# coding.md

> 适用范围：本项目及其后续同风格代码开发。
> 目标：让代码在**可读性、稳定性、一致性、可维护性、可复现性**上长期保持高质量。
> 优先级：**正确性 > 清晰性 > 一致性 > 性能优化 > 炫技式写法**。

---

# 1. 总体原则

## 1.1 先追求正确和清晰，再追求复杂能力
- 不为“看起来高级”而引入复杂结构。
- 任何抽象都必须服务于当前或高度确定的近未来需求。
- 如果一个实现让后来的维护者需要停下来猜含义，那它就还不够好。

## 1.2 代码首先是写给人看的，其次才是给机器执行的
- 命名必须表达业务语义。
- 目录结构必须能体现模块边界。
- 注释必须解释“为什么这样做”，不是翻译代码字面意思。

## 1.3 单一来源、单一真相
- 不允许多套旧逻辑与新逻辑长期并存。
- 不允许同一配置、同一常量、同一数据定义散落多处。
- 对外接口、类型定义、配置项要尽量集中管理。

## 1.4 小步快跑，随改随验证
- 每次改动都应尽量保持范围可控。
- 每次实质性改动后，要至少进行与改动范围匹配的验证。
- 修 bug 时优先先复现，再修复，再验证。

# 2. 目录与文件组织规范

## 2.1 目录设计原则
- 按“业务模块/职责”拆分，不按“文件类型堆放”拆分。
- 同一模块下，相关逻辑尽量内聚。
- 公共代码只有在被多个模块真实复用时才抽离。

## 2.2 建议目录特征
一个清晰项目至少应当让人快速找到：
- 页面或入口
- API/服务层
- 数据模型/类型
- 公共组件
- AI 能力封装
- 配置项
- 测试
- 脚本
- 文档

## 2.3 文件长度控制
- 单文件过长时优先拆分。
- 一般原则：
  - 超过 300~500 行的业务文件要评估是否可拆。
  - 超过 1000 行通常意味着职责混乱。
- 拆分要按职责拆，不要机械切割。

## 2.4 文件命名
- 文件名应体现用途，不使用含糊命名。
- 避免：
  - `utils.ts`
  - `common.py`
  - `temp.js`
  - `test_new_final_v2.py`
- 推荐：
  - `lesson-pack-generator.ts`
  - `student-qa-service.ts`
  - `analytics-report-builder.py`

# 3. 命名规范

## 3.1 通用要求
- 名字必须表达真实含义。
- 名字要让人能看出：它是什么、做什么、服务谁。
- 避免无意义缩写，除非是领域内公认缩写。

## 3.2 变量命名
- 使用完整、可理解的业务语义命名。
- 避免：`data`, `info`, `tmp`, `obj`, `res`, `ret` 作为长期变量名。
- 循环中的临时变量可以短，但业务核心变量不可以。

## 3.3 函数命名
- 函数名应体现动作。
- 推荐使用动词开头：
  - `buildLessonPack`
  - `saveCourseProfile`
  - `analyzeStudentQuestions`
- 判断类函数用：
  - `is...`
  - `has...`
  - `can...`

## 3.4 类/模块命名
- 类名用名词或名词短语。
- 模块名体现领域职责，而非底层技术细节。

## 3.5 布尔值命名
- 布尔值必须读起来像一个判断。
- 推荐：
  - `isPublished`
  - `hasEvidence`
  - `canRegenerate`
- 避免：
  - `publishFlag`
  - `statusOk`

# 4. 函数与模块设计规范

## 4.1 单一职责
- 一个函数只做一件清晰的事。
- 一个模块只负责一个核心领域。
- 如果一个函数同时做“校验 + 查询 + 格式化 + 落库 + 日志输出”，通常说明设计过重。

## 4.2 函数长度
- 优先写短函数，但不机械追求极短。
- 当一个函数难以一眼说清“它的唯一职责”时，应考虑拆分。

## 4.3 参数设计
- 参数过多时优先改为对象参数。
- 关联参数应成组出现，避免长串位置参数。
- 可选参数要有清晰默认值与行为说明。

## 4.4 返回值设计
- 返回值结构要稳定、可预期。
- 不要让调用方猜测返回类型。
- 对复杂输出优先使用结构化对象/JSON。

## 4.5 副作用控制
- 明确哪些函数有副作用：写文件、落库、发请求、更新状态。
- 纯函数与副作用函数尽量分离。

# 5. 错误处理规范

## 5.1 错误不能吞
- 不允许无声失败。
- 不允许广泛 `try/catch` 后直接忽略。
- 出错时要么修复、要么上抛、要么记录并明确返回失败信息。

## 5.2 错误信息要可定位
- 错误信息至少应包含：
  - 哪个模块
  - 哪个操作
  - 什么失败了
  - 关键上下文是什么
- 避免只有 `something went wrong` 这类无效报错。

## 5.3 面向用户与面向开发者的错误要区分
- 用户看到的错误要简洁、可理解。
- 日志或调试信息要保留技术细节。

## 5.4 警告也要处理
- lint warning、类型 warning、弃用警告，不应长期堆积。
- 能修就修，不要把 warning 当正常状态。

---

# 6. 类型与数据结构规范

## 6.1 能显式就不要隐式
- 关键数据结构必须有明确类型定义。
- 边界输入、接口响应、配置对象、AI 输出结构都要有明确 schema/type。

## 6.2 不信任外部输入
- 所有来自用户、文件、接口、模型输出的数据都要做校验。
- 不要假设外部输入天然合法。

## 6.3 AI 输出必须结构化约束
- 能输出 JSON 就不要只输出大段自然语言。
- 对 AI 输出要做：
  - 格式约束
  - 字段校验
  - 兜底处理

## 6.4 保持数据模型稳定
- 不要频繁随意改动核心数据字段名。
- 若确需变更，要同步更新调用方、文档和迁移逻辑。

---

# 7. 注释与文档规范

## 7.1 注释只写必要信息
注释应该回答以下问题之一：
- 这段代码为什么存在？
- 为什么采用这个方案而不是另一个方案？
- 这里有什么容易误解的边界？
- 这里依赖了什么约束？

## 7.2 不写废话注释
避免：
- `// set value`
- `// loop through items`
- `# initialize variable`

## 7.3 模块级文档
复杂模块建议在文件头说明：
- 模块职责
- 输入输出
- 关键依赖
- 风险点

## 7.4 项目文档与代码同步
- 代码变化后，claude.md / work.md / README / API 文档应及时更新。
- 过时文档比没有文档更糟糕。

---

# 8. 测试与验证规范

## 8.1 每次修改都要验证
- 小改动：至少本地运行相关功能。
- 中改动：补充单测/集成验证。
- 大改动：至少完成一条端到端主流程验证。

## 8.2 优先测试关键路径
优先保证：
- 主流程可用
- 边界输入可控
- 错误场景可解释
- 回归不破坏已有核心能力

## 8.3 测试文件要有语义
- 测试名称要能说明行为，不要只写 `test1` / `works`。

## 8.4 修 bug 要有对应验证
- 一个 bug 被修复后，应至少有一个对应测试、断言或稳定复现步骤。

---

# 9. 重构规范

## 9.1 什么时候应该重构
- 重复逻辑已明显超过两处
- 函数/模块职责混乱
- 命名误导
- 新功能无法自然接入旧结构
- 旧实现显著阻碍理解或测试

## 9.2 重构时要做什么
- 先明确当前行为
- 再调整结构
- 最后验证行为不变

## 9.3 重构不是堆新代码
- 重构后应减少复杂度，而不是引入更多包装层。
- 新实现替换旧实现后，应删除无用旧代码。

---

# 10. 性能与资源使用规范

## 10.1 没有证据不要过早优化
- 不要为了“可能更快”而牺牲可读性。
- 性能优化要基于真实瓶颈。

## 10.2 对高成本操作要有意识
包括但不限于：
- 大文件解析
- 向量检索
- 大模型调用
- 重复网络请求
- 大量磁盘写入

## 10.3 明确缓存边界
- 需要缓存时要说明缓存键、失效机制、刷新条件。
- 不要引入不透明缓存导致结果难以解释。

---

# 11. 日志规范

## 11.1 日志用于定位问题，不是制造噪音
- 关键流程要有日志。
- 高频低价值日志要克制。

## 11.2 日志内容建议
至少包含：
- 模块名
- 操作名
- 结果状态
- 关键上下文

## 11.3 禁止泄露敏感信息
- 不在日志中打印密钥、token、隐私数据、完整大段原文材料。

---

# 12. 配置与环境规范

## 12.1 配置外置
- 环境相关内容必须放入配置或环境变量，不要硬编码。

## 12.2 配置命名清晰
- 配置名必须表达用途。
- 对默认值与缺省行为要明确。

## 12.3 开发/测试/生产要可区分
- 不同环境的行为差异要可控、可解释。

---

# 13. Git 规范

## 13.1 小步提交
- 每个 commit 应围绕一个清晰主题。
- 不要把无关改动揉进同一个提交。

## 13.2 commit message 要可读
推荐风格：
- `feat: add lesson pack generation flow`
- `fix: handle empty material parsing result`
- `refactor: simplify student qa service`
- `docs: update claude and work docs`

## 13.3 提交前检查
提交前至少检查：
- `git status`
- 是否混入无关文件
- 是否存在调试残留
- 是否有明显可删文件

## 13.4 大文件与结果文件不上库
- 权重、数据集、缓存、日志、推理结果、临时导出文件都不要入库。

---

# 14. 删除与清理规范

## 14.1 允许删除无用代码
以下内容应积极清理：
- 被替代的旧实现
- 永不再调用的工具函数
- 已废弃页面
- 失败实验残留脚本
- 临时调试文件
- 过时文档

## 14.2 删除前要确认影响范围
- 是否仍被引用
- 是否影响复现
- 是否是用户要求保留的内容

## 14.3 可疑文件先列出，再处理
- 对结果文件、日志、失败产物等，先向用户报告可清理项，不要直接删除可能仍有参考价值的文件。

---

# 15. 前端代码规范（适用于 React / Next.js）

## 15.1 组件职责清晰
- 页面组件负责组织页面。
- 业务组件负责业务展示。
- 基础组件保持通用。

## 15.2 不要把所有逻辑塞进页面文件
- 请求、状态整理、格式化逻辑要适度拆分。

## 15.3 UI 状态要完整
至少考虑：
- loading
- empty
- error
- success

## 15.4 表单处理
- 字段要有默认值与校验。
- 提交时要明确错误反馈。

## 15.5 样式一致性
- 样式方案统一。
- 不要混用多套风格系统。
- 类名、间距、字号、颜色语义保持一致。

---

# 16. 后端代码规范（适用于 Node / Python 服务）

## 16.1 路由层薄、服务层清晰
- 路由只做参数接收、校验、调用服务、返回响应。
- 核心业务逻辑放在 service/usecase 层。

## 16.2 数据访问与业务逻辑分离
- 不要把数据库操作散落在所有地方。
- 数据层接口要清晰。

## 16.3 外部调用要可替换
- 模型调用、检索调用、文件解析调用尽量封装成独立适配层。

## 16.4 API 响应规范
- 响应结构一致。
- 错误码/错误信息可理解。
- 不要让前端猜字段。

---

# 17. AI / RAG 相关规范

## 17.1 提示词不是魔法
- 先控制输入上下文，再优化提示词。
- 不把所有问题都指望 Prompt 解决。

## 17.2 检索优先于幻想
- 有知识边界的问题优先检索再生成。
- 没证据时明确说明不确定，而不是强答。

## 17.3 输出必须可校验
- 课时包、分析报告、问答证据都要可结构化检查。

## 17.4 模型调用要留痕
- 至少能知道调用了哪个模型/模式、输入输出大致边界、失败原因。

---

# 18. 文档维护规范（必须执行）

## 18.1 claude.md
- 描述当前真实代码结构与实现，不写空泛愿景。
- 代码结构变了就更新。

## 18.2 user.md
- 只记录长期有用的信息：偏好、常用命令、稳定 workaround。
- 不写一次性噪音。

## 18.3 work.md
- 每轮都更新。
- 记录进展、问题、清理建议、下一步。
- 发现旧记录失真时要修订，不是只追加。

---

# 19. 禁止事项

严禁：
- 为了赶进度堆砌不可维护代码
- 明知报错仍假装完成
- 大量复制粘贴重复逻辑
- 留下未说明用途的临时文件
- 提交结果文件、缓存、日志、权重、大数据
- 使用含糊命名隐藏复杂性
- 用注释掩盖糟糕设计而不是修正设计
- 文档长期与代码脱节

---

# 20. 每轮开发完成前的检查清单

在回复用户之前，至少完成以下检查：

1. 当前改动是否只围绕本任务目标？
2. 是否引入了过时/重复/无用代码？如果有，是否已清理？
3. 是否有报错、warning、lint、类型问题未处理？
4. 是否完成了与改动匹配的运行验证/测试？
5. 是否更新了 claude.md、user.md、work.md？
6. 是否检查了可清理文件并向用户汇报？
7. 是否检查了 git status？
8. 是否只提交了核心必要文件？
9. 是否已经 commit？
10. 若条件允许，是否已经 push？

---

# 21. 最终标准

高质量代码应同时满足：
- 别人能快速看懂
- 自己一周后还能放心修改
- 出问题时容易定位
- 新需求能自然接入
- 文档与代码一致
- 仓库整洁，没有大量历史垃圾

如果某个实现虽然“能跑”，但会显著伤害以上目标，则它还不算完成。
