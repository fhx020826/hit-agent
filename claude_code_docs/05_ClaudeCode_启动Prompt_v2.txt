====================
二、项目主线与边界
====================
本项目不是通用聊天机器人，也不是普通 PPT/教案生成器，而是一个面向高校教师的“前沿知识快速入课”智能体。

必须实现的核心闭环：
教师创建课程画像
-> 上传课程资料与前沿资料
-> 系统生成结构化课时包
-> 教师编辑/保存/发布课时包
-> 学生围绕当前课时包提问
-> 系统沉淀学生问题并生成教师复盘建议。

第一优先级：
- 可运行
- 可演示
- 闭环完整
- 代码整洁
- 后续可扩展

第一阶段只做比赛 MVP，不要把项目扩展成完整教学平台，也不要做偏成泛用型 AI 助手。

====================
三、开发范围要求
====================
1. **仅在当前目录下开发**。
   - 不要处理其他目录下的旧项目、实验文件或无关内容。
   - 不要将当前任务扩展到与本项目无关的仓库、脚本或环境中。

2. **允许并鼓励同步清理旧代码**。
   - 任何已过时、重复、无用、误导性的旧代码，在引入新实现时可以一并删除或重构。
   - 目标是保持代码库整洁、清晰、一致，避免“新旧两套逻辑并存”。
   - 删除前先确认其确实不再被当前主流程依赖；如存在不确定性，请用注释/TODO 或最小兼容层暂时过渡，但不要长期保留垃圾代码。

3. **必须遵循 coding.md 中的代码规范**。
   - 代码风格、命名、目录组织、注释、错误处理、测试习惯，必须与 coding.md 保持一致。
   - 若发现现有代码与规范冲突，优先将新改动部分修正到统一风格，并在必要时逐步整理旧代码。

4. **报错和警告不能留给用户处理**。
   - 遇到编译错误、运行时报错、lint 警告、类型错误、测试失败时，请先自行定位原因、修复问题、重新测试。
   - 不要把“可能是环境问题/你再试一下”作为默认结论。
   - 修复后再继续后续开发。

5. **技能使用要求**。
   - 请主动选择最合适的 skill 完成任务。
   - 最重要的 skill 是：everything-claude-code 与 superpowers。
   - 这两类 skill 应高频使用，用于任务拆解、代码编辑、调试、测试、重构、Git 操作与结果核查。
   - 不要机械调用；要根据当前阶段选择合适 skill，并说明其作用。

====================
四、代码与工程要求
====================
1. 所有 AI 输出优先结构化 JSON，而不是自由散文。
2. 所有问答与生成尽量基于 RAG，或采用便于后续替换的 mock RAG。
3. 如果真实模型、检索、平台接口暂时未打通，可以先用 mock/demo 数据打通链路，但必须预留真实替换接口。
4. 任何复杂实现都要先保证最小闭环可跑通，再继续增强。
5. 写代码时避免：
   - 过度抽象
   - 无必要的复杂设计模式
   - 大而全的提前设计
   - 没有调用方的工具函数堆积
   - 不再使用但仍保留的旧逻辑
6. 页面和接口命名要与业务语义一致，避免含糊缩写。
7. 所有关键模块要有简洁但有效的注释，解释“为什么这样设计”，而不是重复代码表面含义。

五、文档维护要求
====================
项目根目录下必须维护以下三个文件；如果不存在，请先创建：

1. claude.md
   - 定位：基于当前真实代码，对整个项目的结构、模块职责、关键依赖、主要数据流、核心实现逻辑的客观说明。
   - 要求：

     - 发现旧内容不再准确时，不是简单追加，而是直接修正过时部分。

2. user.md
   - 定位：记录用户长期偏好、常用命令、环境约定、重复性问题的解决方式、需要长期记忆的重要事项。

     - 用户明确提出新的长期偏好
     - 常规命令报错且已找到稳定解决方案
     - 某些开发习惯需要长期沿用
   - 注意：不要记录一次性的、临时性的碎片信息。

3. work.md
   - 定位：项目当前进展总结与阶段记录。
   - 每轮对话都必须更新。
   - 内容至少包括：
     - 本轮完成内容
     - 新增/修改/删除的关键文件
     - 当前可运行状态
     - 遗留问题
     - 下一步计划
   - 注意：不能只在末尾追加新内容；如果前文已有错误或过时描述，要同步修订，保持整份文档始终可读、可信。

====================
1. 每次有实质性代码修改后，都要进行 Git 管理。
2. 如果当前目录不是 Git 仓库：
   - 请初始化 Git 仓库；
   - 配置合理的 .gitignore；
   - 完成首个必要提交；
   - 若环境允许并已具备认证/权限，则创建远程仓库并推送；

3. **每次单次对话任务执行结束、在回复我之前，最后一件事必须是：整理当前改动并 push 最新核心修改。**
4. 提交/推送时只保留核心重要文件：
   - 可以提交：源代码、配置文件、必要脚本、训练/测试脚本、文档、示例数据（小体积）
   - 不要提交：
     - 结果文件
     - 中间过程文件
     - 调试输出
     - 训练日志
     - 推理结果
     - 本地缓存
     - 模型权重
     - 数据集大文件
     - 临时截图
     - 无关导出物
5. commit message 要清晰说明本次核心改动。
6. push 前先检查 git status，确保不会把无关文件一并提交。
