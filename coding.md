# 编码规范

## 1. 总原则
- 所有开发、运行、调试默认在 `fhx-hit-agent` conda 环境中进行。
- 所有项目相关操作仅在 `/home/hxfeng/fhx-hit-agent` 内进行。
- 代码、文档、接口说明必须与真实实现一致，不允许写出“看起来有但实际没落地”的内容。
- 优先保持目录整洁。过时脚本、失败实验结果、无价值中间日志、重复文档要及时识别。

## 2. 代码风格
- 默认使用 ASCII；只有文件本身已使用中文或确有必要时才保留中文文本。
- 注释统一使用中文，要求简洁、直接、解释“为什么”，避免解释显然代码。
- 命名优先清晰可读，避免缩写堆叠。
- 修改时优先小步、局部、可验证，避免一次性大范围重写。

## 3. Python 约定
- 进入环境：
  - `eval "$(/home/hxfeng/miniconda3/bin/conda shell.bash hook)" && conda activate fhx-hit-agent`
- 新增依赖前先确认是否已有等价依赖，避免环境膨胀。
- 涉及路径、模型、数据文件时优先使用相对项目根目录的组织方式。
- 运行前尽量先做最小导入检查，再做服务启动或长任务执行。

## 4. 前后端协同
- 前端展示字段、文案、状态值要与后端真实接口保持一致。
- 接口变更时同步检查：
  - 前端类型定义
  - 页面使用点
  - README / docs
  - `project.md`
  - `work.md`

## 5. 文档规范
- 每轮对话后更新 `work.md`，记录真实进展、阻塞、下一步。
- 当项目理解发生变化时更新 `project.md`，不要只追加新内容，过时部分要同步修正。
- 用户习惯、常用命令、环境坑位更新到 `user.md`。
- 长期规范和开发约束维护在本文件。

## 6. Git 约定
- 有实际修改后及时提交，提交信息要能说明本次变更主题。
- 重要代码和文档更新尽量同步到远程仓库。
- 当前 remote 约定：
  - `origin` = `fhx020826/hit-agent`
  - `upstream` = `wishmyself/hit-agent`
- 每次开始开发前先执行远程同步检查，避免在过期代码上继续修改：
  - `bash -ic 'clash && proxy && cd /home/hxfeng/fhx-hit-agent && git fetch --all --prune'`
  - `git pull --rebase origin main`
- 如需吸收原始团队仓库更新，再执行：
  - `git fetch upstream`
  - `git rebase upstream/main`
- 不上传以下内容：
  - 大型测试输出
  - 训练/推理日志
  - 中间结果
  - 数据集
  - 模型权重
  - 本地缓存

## 7. 清理原则
- 发现可清理内容时，先在对话中明确告知用户，再删除。
- 对已验证失败且无参考价值的实验输出，可建议清理，但不自行直接删除。
- 避免新增重复说明文档；已有文档可更新时不新建平行版本。
